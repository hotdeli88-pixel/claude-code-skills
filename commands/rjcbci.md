# 전북형 CBCI 단원설계 검증·개선·재검증 (rjcbci)

## 개요

전북형 개념기반 탐구학습(CBCI) 단원설계 HWPX 문서를 **3단계 파이프라인**(검증 → 개선 → 재검증)으로 처리한다. Phase 1에서 5개 전문가 에이전트가 병렬 검증, Phase 2에서 발견된 이슈를 통합 개선하여 HWPX에 반영, Phase 3에서 개선 결과를 재검증한다.

```
Phase 1                                Phase 2        Phase 3
                                        (순차)         (병렬 2개)
 ┌─ B: 일반화 설계 ─┐
 │  (foreground)    │
 └────────┬────────┘
          ↓
   사용자 컨펌/수정 ✋
          ↓
 ┌─ B2: 수정 재검증 ─┐  (수정 시에만)
 │  [+ NLM 소스참조] │
 └────────┬──────────┘
          ↓ PASS? (최대 3회)
 ┌─ A: 모형 구조  ─┐    ┌────────┐    ┌─ G: 구조재검증 ─┐
 ├─ C: 탐구질문   ─┼──→ │ F: 통합 │──→ │                │
 ├─ D: 평가 체계  ─┤    │  개선   │    │                │
 └─ E: 탐구기법   ─┘    └────────┘    └─ H: 내용재검증 ─┘
   (병렬 4개)
```

## 사용법

```
/rjcbci <HWPX 파일 경로>
/rjcbci ./고등학교_수학_사례_v3.hwpx
/rjcbci ./단원설계.hwpx ./전북형_템플릿.pdf ./루브릭.png
```

## 입력 파일

| 파일 | 필수 | 용도 |
|------|------|------|
| HWPX 단원설계 | 필수 | 검증 대상 문서 |
| 공식기준 PDF | 선택 | 전북형 CBCI 템플릿(안). 없으면 내장 기준 사용 |
| 루브릭 PNG | 선택 | 평가 루브릭 양식 이미지. 있으면 평가 체계 검증에 참조 |

| NotebookLM notebook_id | 선택 | `--nlm` 인자로 전달. CBCI 소스가 업로드된 노트북 ID. 있으면 Agent B/B2가 소스 기반 검증 수행 |

## 공식 기준 (내장)

### 탐구학습 모형 3가지

#### 발견중심형
상위범주: 개념탐색 → 일반화 진술 및 적용
① 사실과 주제 탐색 (탐구질문 만들기)
② 개념 속성 파악, 정의 (사례 속 공통 특징 탐색)
③ 일반화 문장 진술하기 (조사·조직·정리 바탕)
④ 일반화 적용 및 평가 (탐구질문 연계 평가)

#### 원리탐구형
상위범주: 일반화탐색 → 개념명료화 및 일반화적용
① 일반화 문장 살펴보기
② 생활 속 일반화 관련 상황, 예시 등, 탐구 질문 만들기
③ 개념 탐색 및 정의 (일반화 문장 적용 사례 탐구, 개념 명확히)
④ 일반화 적용 및 평가하기 (탐구질문의 답 구성 및 평가)
**핵심**: 일반화를 먼저 제시 → 사례로 내려가는 **연역적** 접근

#### 과정구성형
상위범주: 기능 및 전략 탐색 → 전략 적용 및 일반화
① 문제 인식 및 탐구 질문 도출 (예시에서 목표 전략 파악, 사실적·개념적 질문 만들기)
② 전략(기능, 개념) 파악하기 (문제 해결을 위한 준비)
③ 전략 탐구하여 문제 해결하기 + 평가 (기능·전략을 새로운 상황에)
④ 일반화하기 (탐구질문의 답 구성)

### 학생상 (5가지, 1개 선택)
1. 새로움을 제안하고 대안을 마련하는 학생
2. 자신의 강점을 찾고 진로를 탐색하는 학생
3. 공감하고 배려하며 상식을 지키는 학생
4. 자신의 몸과 마음을 이해하고 관리하는 학생
5. 서로 다름을 알고 다양함을 인정하는 학생

### 핵심 가치 (13개 중 1~2개 선택)
진취성, 창의성, 혁신성, 주도성, 의사소통과 협력, 비판적 사고와 문제해결, 디지털 리터러시, 책임과 안전, 리더십, 사고의 유연성, 적응성, 다양성, 자기 표현

### 핵심 개념 (26개 중 2~3개 선택)
원인과 결과, 시스템, 주기, 상호작용, 다양성, 갈등과 협력, 패턴, 변화, 심미성, 공감과 소통, 공동체, 평등, 인권, 연결, 문화, 형태, 관점, 논리, 관계, 시간과 공간, 창의성, 자유, 공정, 정보, 기능, 책임

### 핵심 기능 (9개)
설명하기, 해석하기, 분석하기, 적용하기, 공감하기, 관점갖기, 종합하기, 평가하기, 창안하기

### 탐구질문 유형 및 배치
- **사실적(사)**: 관찰 가능한 구체적 데이터·수치·현상을 직접 묻는 질문
- **개념적(개)**: 관계·패턴·원인·의미를 탐구하는 질문
- **논쟁적(논)**: 가치 판단·해석·관점 대립·선택을 요구하는 질문
- 배치: 소주제1=(사+개만), 소주제2=(사+개, 논 제한적), 소주제3=(사+개+논)

### 일반화 요건
- 보편성: 교과를 넘어 적용 가능
- 추상성: 구체적 행위가 아닌 관계·원리 진술
- 초월성: 특정 단원 내용을 넘어선 전이 가능한 통찰
- 교과특수용어(예: "이차방정식", "b²-4ac", "x축" 등)가 주어·목적어에 없어야 함

### 3차원 평가
- 1차원(이해): 개념의 이해 확인
- 2차원(적용): 소주제/단원 일반화 이해 여부 확인
- 3차원(전이): 단원 일반화를 넘어선 전이·실행 평가

### Step 1: Phase 1 — 검증 (Agent B 선행 + 나머지 4개 병렬)

**실행 순서가 중요하다:**

1. **먼저 Agent B만 foreground로 실행**한다. (일반화 제안 → 사용자 컨펌 필요)
2. Agent B 결과를 사용자에게 제시하고 **일반화 컨펌을 받는다**.
3. **사용자가 수정한 경우**: Agent B2(재검증)를 foreground로 실행한다.
   - B2 PASS → Step 4로
   - B2 FAIL → 피드백 제시 + 사용자 재수정 (최대 3회, 이후 현재안 확정)
4. **일반화 확정 후**: 컨펌된 일반화를 기준으로 **Agent A/C/D/E 4개를 병렬(background)로 실행**한다.
   - 컨펌된 일반화가 있어야 탐구질문 정합성(C), 평가 연결(D) 판정이 가능하므로.

각 에이전트에게 Step 0에서 추출한 전체 텍스트 데이터 + 공식 기준을 프롬프트에 포함하여 전달한다.

#### Agent A — CBCI 모형 구조 전문가
```
검증 영역:
- 각 소주제의 학습 모형(발견중심형/원리탐구형/과정구성형)별 탐구단계 순서·명칭
- 원리탐구형 ①②순서 도치 여부 (연역적 접근 핵심 검증)
- 과정구성형 ③④ 분리 여부
- 차시 배분 적절성
출력: 소주제별 PASS/FAIL + 권고사항
```

#### Agent B — 핵심개념·일반화·학생상 전문가 ⚠️ 사용자 컨펌 필요
```
검증 영역:
- 학생상: 공식 5대 학생상 중 하나인지
- 핵심 개념(주): 공식 26개 중 하나인지 + 단원 정합성
- 핵심 기능: 공식 9개 중 선택했는지 + 활동 정합성
- 핵심 가치: 공식 13개 중 선택했는지 + 수량(1~2개)
- 일반화 진술: 보편성·추상성·초월성 3요건 + 교과특수용어 잔존
- 소주제 일반화 → 단원 일반화 위계 정합성

일반화 설계 (PASS/FAIL 무관하게 항상 수행):
- 성취기준 + 핵심개념 + 소주제 구조를 종합 분석하여 "이 단원의 본질적 통찰"을 도출
- 핵심개념이 일반화 문장의 주어·술어에 유기적으로 반영되도록 설계
- 소주제별 일반화가 단원 일반화의 하위집합이 되는 위계 구조로 설계
- 설계 절차:
  ① 성취기준에서 교과 내용 핵심 파악
  ② 핵심개념 렌즈로 교과 초월적 관계·원리 추출
  ③ "~하면 ~가 드러나고/변화하며/결정된다" 형식으로 진술
  ④ 다른 교과에서도 성립하는지 초월성 검증 (예: 과학, 사회, 예술 적용)
  ⑤ 소주제별 일반화 → 단원 일반화 위계 구성
  ⑥ [NotebookLM 소스 참조] (--nlm 인자가 있을 때만 수행)
     - /home/sdm24/notebooklm-server/venv/bin/python /home/sdm24/notebooklm-server/nlm_helper.py query <notebook_id> "이 단원의 핵심 일반화는 무엇이어야 하는가? 성취기준: [성취기준], 핵심개념: [핵심개념]"
     - 소스 기반 답변을 일반화 설계의 추가 근거로 활용
     - 출력에 "NLM 소스 참조" 항목 추가
- 기존 일반화와 제안 일반화를 나란히 비교하는 표 제시

출력:
  (1) 요소별 PASS/FAIL + 교과특수용어 목록
  (2) 제안 일반화 (단원 + 소주제별) — 항상 제시
  (3) 기존 vs 제안 비교표
```

**⚠️ Agent B는 다른 에이전트와 달리 background가 아닌 foreground로 실행한다.**

Agent B 결과가 도착하면 즉시 사용자에게 다음을 제시하고 컨펌을 받는다:

```
## 일반화 진술 제안 — 컨펌 필요

### 단원 일반화
| | 내용 |
|---|---|
| 기존 | (현재 문서의 일반화) |
```

### Step 2: Phase 2 — 통합 개선

Agent A~E의 결과를 수집한 후:

1. **문제 우선순위 분류**
   - 긴급: 구조적 오류 (모형 단계 순서, 질문 배치 위반)
   - 중요: 품질 이슈 (일반화 교과특수용어, 성찰 누락, 루브릭 미구체화)
   - 권고: 완성도 향상 (기법 다양성, 기능 수량 조정)

2. **구체적 개선안 작성** (수정 전 → 수정 후 테이블)

3. **HWPX 반영** — Python 스크립트로 section0.xml의 텍스트 노드 치환:
   ```python
   import zipfile, xml.etree.ElementTree as ET
   # 1. ZIP 해제 → section0.xml 읽기
   # 2. 텍스트 노드 찾기 → 치환
   # 3. 수정된 XML로 새 HWPX 패키징
   ```

   **치환 가능 항목** (텍스트 노드 단위):
   - 일반화 진술 교체
   - 핵심 질문 교체
   - 질문 유형 레이블 변경 (논→개)
   - 이동 표시 태그 부착 [→소주제N 이동]
   - 탐구기법 텍스트 추가 (성찰 활동, 루브릭 등)
   - 단계명 변경
   - GRASPS 텍스트 추가

   **수동 편집 필요 항목** (XML 구조 변경):
   - 테이블 행 순서 변경 (row swap)
   - 질문의 실제 셀 이동 (셀 추가/삭제)
   → 이 항목은 `수동_편집_필요.md`에 기록

4. **산출물 저장**:
   - `Phase1_검증보고서.md` (5개 에이전트 결과 통합)
   - `<원본명> v(N+1).hwpx`

## 주의사항

1. **에이전트 프롬프트에 공식 기준 전문을 포함**할 것. 에이전트는 별도 파일을 읽을 수 없으므로 프롬프트에 기준 목록(학생상 5개, 가치 13개, 개념 26개, 기능 9개, 모형별 단계, 질문 배치 기준)을 모두 포함해야 한다.

2. **HWPX 텍스트 추출은 Step 0에서 1회만** 수행하고, 추출 결과를 에이전트들에게 전달한다.

3. **XML 테이블 행 순서 변경(row swap)은 텍스트 치환으로 불가**하므로, 단계명만 치환하고 활동 내용 교환은 수동 편집 항목으로 기록한다.

4. **Phase 1 에이전트는 반드시 `run_in_background: true`로 실행**하여 병렬성을 확보한다. 5개 에이전트가 모두 완료된 후 Phase 2로 진행한다.

5. 원본 HWPX는 수정하지 않는다. 항상 새 버전 파일을 생성한다.

6. **NotebookLM 연동은 Step 0.5에서 대화형으로 결정**한다. 사용자가 notebook_id를 입력하면 연동하고, 건너뛰면 NLM 없이 진행한다. 인증 만료 시 `nlm login` 실행을 안내하고, 로그인 완료 또는 건너뛰기를 기다린다. NLM 연동 실패 시에도 검증 워크플로우는 중단하지 않는다.
